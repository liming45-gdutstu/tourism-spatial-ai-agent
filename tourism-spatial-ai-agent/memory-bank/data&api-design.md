# 数据与接口设计文档

## 总述

本文档为「基于空间感知的AI个性化文旅规划Agent」专用数据结构、接口入出参规范，严格匹配architecture.md五层分层架构，所有GIS代码、Coze插件、RAG知识库的数据交互均遵循本文表格定义，无任何可执行业务代码。

---

## 一、用户需求入参结构体（大模型解析层输入）

| 字段名            | 数据类型     | 字段含义             | 取值范围/说明                                    |
| ----------------- | ------------ | -------------------- | ------------------------------------------------ |
| target_city       | string       | 目标游玩城市         | 国内地级市/旅游城市名称，如广州、杭州            |
| travel_days       | int          | 游玩总天数           | 1~7天                                            |
| crowd_type        | string       | 出行人群类型         | 青年单人、亲子家庭、中老年、情侣                 |
| travel_season     | string       | 出行季节             | 春季、夏季、秋季、冬季、法定节假日               |
| walk_tolerance    | int          | 单日最大步行耐受距离 | 青年≤12km；亲子≤6km；中老年≤4km                  |
| crowd_sensitivity | string       | 客流拥挤耐受度       | 低（避开热门景点）、中（均衡）、高（无所谓拥挤） |
| scenic_preference | list[string] | 景点偏好标签         | 自然风光、人文古迹、美食街区、网红打卡、博物馆   |

---

## 二、POI景点数据表（RAG知识库底层存储结构）

| 字段名            | 数据类型 | 字段含义             | 取值范围/说明                                |
| ----------------- | -------- | -------------------- | -------------------------------------------- |
| poi_id            | string   | 景点唯一编号         | 高德POI原始ID                                |
| name              | string   | 景点名称             | 中文景点全称                                 |
| lng               | float    | 经度                 | 城市地理经度坐标                             |
| lat               | float    | 纬度                 | 城市地理纬度坐标                             |
| scenic_type       | string   | 景点分类             | 对应用户偏好标签                             |
| avg_traffic_min   | int      | 相邻点位平均通勤时长 | 单位：分钟，用于通达度权重计算               |
| season_popularity | dict     | 分季节客流热度       | key=季节，value=热度值0~10（数值越高越拥挤） |
| cluster_id        | int      | 片区聚类编号         | Kmeans算法输出，同编号为同一游玩片区         |

---

## 三、GIS计算输出-行程结果数据表

| 字段名              | 数据类型     | 字段含义             | 取值范围/说明                      |
| ------------------- | ------------ | -------------------- | ---------------------------------- |
| day_index           | int          | 游玩天数序号         | 1~travel_days                      |
| daily_poi_list      | list[poi_id] | 当日游玩景点序列     | 按通勤最优顺序排列                 |
| total_walk_km       | float        | 当日总步行距离       | 单位：km，不超过crowd_type耐受上限 |
| cross_cluster_count | int          | 当日跨片区次数       | 数值越小，行程折返越少             |
| rest_point_num      | int          | 当日推荐休息点位数量 | 根据步行距离自动匹配               |
| congestion_score    | float        | 当日拥挤综合评分     | 0~10，数值越低游玩体验越好         |

## 四、外部接口：高德地图POI调用出入参

### 4.1 高德API入参

| 字段      | 类型   | 说明                     |
| --------- | ------ | ------------------------ |
| city      | string | 目标城市                 |
| keywords  | string | 景点分类关键词           |
| output    | string | 返回数据格式，固定json   |
| page_size | int    | 单次拉取景点数量，固定50 |

### 4.2 高德API出参（提取字段）

仅提取：名称、经纬度、类型、商圈，其余字段丢弃，存入POI数据表。

---

## 五、内部GIS模块输入输出参数规范

### 5.1 GIS整体模块输入

1. 用户完整入参结构体（表一全部字段）
2. 清洗完成的POI全量数据集（表二）

### 5.2 四大子模块专属入出参

#### 模块1：Kmeans片区聚类

- 输入：所有POI经纬度数组
- 输出：每个POI绑定cluster_id片区编号

#### 模块2：交通通达度权重计算

- 输入：同片区POI两两通勤时长
- 输出：景点排序权重值，权重越高越优先相邻游玩

#### 模块3：客流承载力筛选

- 输入：出行季节、拥挤耐受度、景点season_popularity
- 输出：过滤高拥挤景点后的精简POI列表

#### 模块4：分人群步行体力约束

- 输入：单日预估步行距离、人群walk_tolerance阈值
- 输出：删减超限景点，生成符合步行上限的最终景点序列

### 5.3 GIS模块最终统一输出

完整行程结果数据表（表三），同步传递至大模型润色、离线可视化页面。

## 六、文档使用约束

1. 所有Python脚本、Coze插件、页面可视化读取数据，字段名称、数据类型必须和本文表格完全一致，禁止自定义字段；
2. 新增人群类型、景点标签、GIS指标时，同步更新本文档、architecture.md、CLAUDE.md；
3. 开发前必须完整读取本文档，避免数据结构不匹配引发计算报错。
